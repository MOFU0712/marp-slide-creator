## アーキテクチャ上の重要な設計判断

### 1. JSONをsource of truthにする

スライドデータはJSONで管理し、Marp Markdown・SVG・pptxはすべてJSONから生成する。

```
slides[] (JSON)
  ├── → Marp Markdown  （JSON→md変換はフロントエンド）
  ├── → SVG図          （figureパラメータ→SVGはRendererが担当）
  └── → pptx / pdf     （Marp CLI がバックエンドで処理）
```

**理由：** pptxをsource of truthにするとAIが読み書きしにくい。MarkdownベースのMarpはAIフレンドリーだが、図の埋め込み（SVG）がトークンを大量消費する。JSONを軸にすることで両問題を解決する。

### 2. AIはSVGを生成しない

図のAI生成は、ノード・エッジ・スタイルのJSONパラメータのみを返す。SVGへの変換はフロントエンドのRendererが担当する。

```
AI が返すもの:
  { "nodes": [...], "edges": [...], "style": {...} }

AI が返さないもの:
  <svg>...</svg>  ← Renderer が生成する
```

**理由：** SVGはトークン数が多く、微調整のたびに全体を再生成するコストが高い。パラメータのみにすることでトークンを節約し、UIのポチポチ操作で直接調整できる。

### 3. APIキーはユーザーが設定

Claude API キーは各ユーザーが自身のキーを環境変数または設定ファイルで設定する。APIリクエストはNext.js API Route経由で行い、ブラウザには露出させない。

**理由：** ローカル実行のため、各ユーザーが自身のAPIキーを管理する。API Routeを経由することで、ブラウザのDevToolsからキーが見えることを防ぐ。

### 4. SSEでストリーミングレスポンス

骨子生成・スライドAI修正はServer-Sent Eventsで逐次レスポンスを返す。

```
Frontend                     Backend                  Claude API
   │                            │                         │
   │── POST /story/generate ──►│                         │
   │                            │── stream request ──────►│
   │◄── SSE: slide 1 ──────────│◄── token stream ────────│
   │◄── SSE: slide 2 ──────────│                         │
   │◄── SSE: done ─────────────│                         │
```

**理由：** 骨子生成は数秒〜十数秒かかる。最初のスライドが生成された時点でUIに表示し始めることで、体感速度を向上させる。

### 5. エクスポートはローカルファイル出力

pptx・PDF変換はMarp CLIで処理し、ユーザー指定のローカルパスに出力する。

```
POST /api/export → { filePath }  // ローカルファイルパスを返す
```

**理由：** ローカル実行のため、変換結果は直接ファイルシステムに書き出す。ブラウザからはダウンロードリンクを提供。

## 技術的制約と要件

### フロントエンド

- **ブラウザサポート：** Chrome・Firefox・Safari 最新2バージョン
- **モバイル対応：** レスポンシブ対応は行うが、図エディタはデスクトップ操作を前提とする
- **オフライン：** 非対応（AIリクエストにネットワークが必要なため）
- **@marp-core のブラウザ内動作：** Marpのテーマ・CSSはブラウザ内でレンダリングし、サーバー往復なしにプレビューを更新する

### ローカル実行環境

- **ランタイム：** Node.js 20（Next.js + Marp CLI）
- **データ管理：** セッション中はメモリ上に保持、終了時に破棄（永続化なし）
- **素材ファイル：** ローカルファイルシステムから読み込み（ファイルパスまたはディレクトリパス指定）

### セキュリティ

- **APIキー：** Claude API キーは環境変数（`.env.local`）で管理。API Route経由でアクセスし、ブラウザに露出しない
- **ローカル実行：** データはユーザーのマシン外に出ない（ローカルファイルシステムのみ使用）
- **入力バリデーション：** Zod によるリクエストバリデーション
- **XSS対策：** SVGのレンダリングはReactのDOMを経由し、dangerouslySetInnerHTMLを最小限に抑える

## パフォーマンス要件と対策

| 要件 | 目標値 | 対策 |
|---|---|---|
| 骨子生成の初回表示 | 3秒以内 | SSEで最初のスライドを逐次表示 |
| スライドプレビュー更新 | 500ms以内 | @marp-core のブラウザ内レンダリング、デバウンス処理 |
| エクスポート（pptx） | 30秒以内 | Marp CLI でローカル変換 |
| ページ初回ロード | 2秒以内 | Next.js のローカル開発サーバー |

## 段階的リリース計画

### Phase 1（プロトタイプ）
- Next.js フロントエンド + API Route
- Claude API キーは環境変数で設定
- データはセッション中のみメモリ上に保持（永続化なし）
- Marp md エクスポートのみ対応
- 素材はテキスト貼り付けで入力

### Phase 2（機能拡充）
- ローカルファイル/ディレクトリ指定による素材読み込み
- pptx・PDF・HTML エクスポート対応（Marp CLI 統合）
- 図エディタの全機能実装

### Phase 3（安定版）
- エラーハンドリング・ロギングの整備
- E2E テスト整備
- ドキュメント整備・OSS公開準備
