# ユビキタス言語定義

このプロジェクトで使用するドメイン用語・UI用語・コード上の命名を統一する。
会話・ドキュメント・コード・UIのすべてで同じ言葉を使うことで認識のズレを防ぐ。

---

## ドメイン用語

### プロジェクト (Project)

スライド1本分の作業単位。モード・インタビュー回答・テーマCSS・スライド群をまとめたもの。
ユーザーが「1つのスライドを作る」ために作成し、保存・再開できる。

- コード上の命名：`Project`
- DBテーブル：`projects`

### モード (Mode)

スライドのストーリーテンプレートの種類。インタビュー内容とプロンプトのシステム指示がモードによって変わる。

| 日本語 | 英語（コード） | 説明 |
|---|---|---|
| 提案モード | `proposal` | 新技術・新施策の導入を提案するスライド |
| 実装報告モード | `report` | 実装した内容・結果を報告するスライド |
| 技術比較モード | `comparison` | 複数の技術・手法を比較検討するスライド |
| 調査報告モード | `research` | 調査・検証の結果をまとめるスライド |

### 素材 / 素材ドキュメント (Source File)

骨子生成のインプットとしてアップロードするファイル。
実装ドキュメント・調査メモ・既存pptxなどを指す。
「素材を読ませる」「素材から調理する」という表現で使う。

- コード上の命名：`SourceFile`
- DBテーブル：`source_files`

### インタビュー (Interview)

モード選択直後にAIが行う質問フェーズ。3〜4問をフォームで一括回答する。
「AIインタビュー」「インタビューに答える」という形で使う。

- コード上の命名：`InterviewAnswers`

### 骨子 (Outline)

インタビュー回答と素材をもとにAIが生成するスライドの構成案。
各スライドのタイトル・役割・キーメッセージ・推奨図タイプで構成される。
「骨子を生成する」「骨子を確認する」「骨子を差し戻す」という形で使う。
「構成案」「アウトライン」は使わず「骨子」に統一する。

- コード上の命名：`StoryOutline`（生成中）、`Slide.role` + `Slide.keyMessage`（確定後）

### スライド (Slide)

骨子確定後に生成されるプレゼンテーションの1ページ。
テキストコンテンツ・図・スピーカーノート・役割・キーメッセージを持つ。

- コード上の命名：`Slide`
- DBテーブル：`slides`

### 役割 (Role)

そのスライドがストーリー上で果たす目的。骨子フェーズで決定し、スライド編集フェーズまで引き継ぐ。
AIチャットでの修正指示に文脈として使われる。
例：「本番移行の承認を取るためのスライド」

- コード上の命名：`Slide.role`

### キーメッセージ (Key Message)

そのスライドで伝えたい最重要の一文。骨子フェーズで決定する。
例：「全国MAPE 6%を達成、当初目標の8%を上回った」

- コード上の命名：`Slide.keyMessage`

### 図 (Figure)

スライドに埋め込むビジュアル要素。図タイプとパラメータで定義され、SVGはフロントエンドのRendererが生成する。
「図を追加する」「図のパラメータを変更する」という形で使う。
「SVG」とは区別する（SVGは実装の詳細であり、ユーザーが意識するのは「図」）。

- コード上の命名：`Figure`
- DBテーブル：`figures`

### 図タイプ (Figure Type)

図の種類。以下の5種類に限定する。

| 日本語 | 英語（コード） | 用途 |
|---|---|---|
| アーキテクチャ図 | `architecture` | システム構成・コンポーネント関係 |
| フロー図 | `flow` | 処理の流れ・ワークフロー |
| 棒グラフ | `bar_chart` | 数値の比較・精度の可視化 |
| タイムライン | `timeline` | マイルストーン・スケジュール |
| 比較表 | `comparison` | 技術・手法の比較 |

### テーマCSS (Theme CSS)

全スライドに適用するMarp準拠のCSSスタイル定義。
プロジェクトに1つだけ存在する。
「テーマを変更する」「CSSを設定する」という形で使う。

- コード上の命名：`Project.themeCSS`

### エクスポート (Export)

完成したスライドを配布・発表用のフォーマットに出力すること。

| フォーマット | 説明 |
|---|---|
| Marp md | Marp Markdown形式（`.md`）。再編集・バージョン管理に適する |
| pptx | PowerPoint形式。発表環境を選ばない |
| PDF | 印刷・配布用 |
| HTML | ブラウザで閲覧できる共有URL |

## UI用語

### ストーリー設計フェーズ

プロジェクト作成から骨子確定までのフェーズ。
素材アップロード → モード選択 → インタビュー → 骨子レビューの順で進む。

### スライド編集フェーズ

骨子確定後にスライドの内容を仕上げるフェーズ。
3ペインレイアウト（サムネイル一覧 / プレビュー / 編集パネル）で構成される。

### 編集パネル

スライド編集フェーズの右側パネル。4つのタブを持つ。

| タブ名 | 役割 |
|---|---|
| テキスト | タイトル・本文・スピーカーノートの直接編集 |
| 図 | 図タイプ選択・ノード/エッジ定義・スタイル設定 |
| スタイル | レイアウト選択・テーマCSS編集 |
| AI | 自然言語によるスライド修正指示 |

### 図エディタ

図の作成・編集に特化したUI。3ペイン構成（ノード定義 / キャンバス / プロパティ）。
「図エディタを開く」「図エディタで編集する」という形で使う。

### 自動整列

図エディタのノードを方向（横 / 縦）に沿って等間隔に再配置する操作。

### 骨子レビュー

AIが生成した骨子を確認・承認・差し戻しするためのUI。
「骨子をOKにする」「差し戻す」「再生成する」という操作がある。

## 技術用語（エンジニア間で使う）

### Source of Truth

スライドデータの正規の保存場所 = JSON（`slides[]`）。
Marp Markdown・SVG・pptxはすべてJSONから生成される派生物であり、JSONが唯一の編集対象。

### Renderer

図タイプ別のSVG生成モジュール（フロントエンド）。
AIはRendererを呼び出さず、パラメータ（JSON）のみを返す。
Rendererがパラメータを受け取りSVGを生成する。

### SSE（Server-Sent Events）

骨子生成・スライドAI修正でバックエンドからフロントエンドへのストリーミングに使うプロトコル。
「SSEで返す」「ストリーミングで表示する」という形で使う。

### 非同期ジョブ

pptx・PDF変換など時間のかかる処理をジョブIDで管理する仕組み。
`POST /export` でジョブを開始し、`GET /export/:jobId` でステータスを確認する。

## 使わない言葉（言い換え対応表）

| 使わない言葉 | 代わりに使う言葉 | 理由 |
|---|---|---|
| アウトライン | 骨子 | 本プロジェクトの統一語 |
| 構成案 | 骨子 | 同上 |
| SVG | 図 | ユーザーが意識するのは図であり、SVGは実装の詳細 |
| スライドデッキ | プロジェクト | 1つの作業単位はプロジェクトと呼ぶ |
| プレゼンテーション | スライド / プロジェクト | 文脈に応じて使い分ける |
| テンプレート | モード | 本プロジェクトのストーリー構成の種類はモードと呼ぶ |
